## 3. Proposed Solution  

### 3.1 Overview  

A **two‑stage AI pipeline** hosted on Clarifai Cloud:

1. **Stage 1 – Vision‑Language Reasoning (VLM)**  
   *Model:* `mm-poly-8b` (frontier multimodal model).  
   *Function:* Reads the image, interprets it against the textual QA checklist, and returns a **structured JSON** with:  
   - `overall_status` (PASS/FAIL)  
   - `defect_list` (array of detected issues)  
   - `explanation` (human‑readable rationale)  

2. **Stage 2 – Targeted Detection (optional)**  
   *Model(s):*  
   - **Object Detection** – `clarifai‑retail‑bg‑detector` (detects background colour, shadows, stray objects).  
   - **Classification** – `clarifai‑color‑accuracy‑cls` (verifies colour consistency).  
   These models output **bounding boxes / confidence scores** when a visual localization is required (e.g., “shadow present”).  

3. **Orchestration via Clarifai Workflows** – Sequential execution, branching if VLM flags a “requires visual localization” condition.  

4. **Result Delivery** – JSON verdict is sent back to the DAM/ CMS via a **REST webhook**; a lightweight UI dashboard shows per‑image status and aggregate statistics.

### 3.2 Why This Architecture  

| Requirement | How it’s satisfied |
|-------------|--------------------|
| **Complex visual reasoning** (e.g., “does the lighting look natural?”) | VLM (`mm-poly-8b`) excels at answering open‑ended visual questions in text. |
| **Explainability** (need for textual justification) | VLM returns a natural‑language explanation; detection models supply visual evidence when needed. |
| **Scalable throughput** (up to 300 imgs/min peak) | Cloud‑managed GPU pool; auto‑scales the workflow. |
| **Compliance & audit** | Structured JSON + optional bounding boxes provide traceable evidence. |
| **Future‑proofing** | Model Evaluation Phase lets H&C benchmark 2‑3 alternative VLMs (e.g., `open‑clip‑v2`, `LLaVA‑1.5‑8b`) before final lock‑in. |
| **Cost efficiency** | Use VLM for high‑level checks (single inference per image) and lightweight detection only when needed, reducing total GPU seconds. |

## 4. Technical Approach  

### 4.1 Data Requirements & Pre‑processing  

| Step | Description |
|------|-------------|
| **Ingestion** | Images pulled from existing DAM via REST API or S3 bucket trigger. |
| **Normalization** | Convert to RGB, resize to max 1024 px (preserves quality, meets VLM input limits). |
| **Metadata Enrichment** | Attach SKU, source, and any existing manual QA label (if available). |
| **Quality Gate** | Validate file integrity (corruption, colour profile) before inference. |

### 4.2 Model Selection  

| Model | Type | Primary Use | Expected Latency (GPU = Standard tier) |
|-------|------|--------------|----------------------------------------|
| `mm-poly-8b` | VLM (frontier) | Textual QA reasoning, classification, explanation | ~1.2 s per 1024×1024 image |
| `clarifai‑retail‑bg‑detector` | Object Detection | Detect background colour deviations, shadows, stray objects | ~0.4 s |
| `clarifai‑color‑accuracy‑cls` | Image Classification | Verify colour match against reference swatches | ~0.2 s |

*Model Evaluation Phase* (Weeks 1‑3) will benchmark these against two alternatives (e.g., `LLaVA‑1.5‑8b`, `open‑clip‑v2`) using a **hold‑out set of 5 k labeled images** from H&C. Decision criteria: accuracy ≥ 90 % on critical defects, latency ≤ 2 s, cost ≤ $0.005 per inference.

### 4.3 Workflow Design  

```mermaid
flowchart TD
    A[Image Ingest (DAM/S3)] --> B[Pre‑process (Resize, Validate)]
    B --> C[Stage‑1: VLM (mm‑poly‑8b)]
    C --> D{VLM Flags Visual Defect?}
    D -->|Yes| E[Stage‑2: Detection Models]
    D -->|No| F[Return PASS JSON]
    E --> G[Aggregate Results]
    G --> H[Webhook / Dashboard]
    F --> H
```

- **Parallelism:** Multiple images are processed concurrently (max batch size = 8) on a shared GPU pool.  
- **Error handling:** Failed inference routes to a dead‑letter queue; admins receive an alert.

## Model Selection Strategy

### Vision-Language Models (VLMs) vs Legacy CV Models

Modern vision-language models significantly outperform traditional computer vision models for most enterprise use cases. We recommend VLMs as the default approach:

| Capability | Vision-Language Models (VLMs) | Legacy CV Models |
|------------|------------------------------|------------------|
| **Understanding** | Comprehends image context and meaning | Pattern matching only |
| **Reasoning** | Can explain WHY something was detected | Binary yes/no output |
| **Flexibility** | Adapt via natural language prompts | Requires retraining |
| **Edge Cases** | Handles novel situations naturally | Fails on unseen patterns |
| **Deployment** | Days (no training needed) | Weeks to months |
| **Maintenance** | Update prompts as needed | Retrain for any change |

### Recommended Models

| Use Case | Recommended Model | Parameters | GPU Tier |
|----------|------------------|------------|----------|
| **Visual Understanding** | mm-poly-8b *(Clarifai)* | 8B | Standard (24GB) |
| **Document Analysis** | Llama 3.2 Vision | 11B | Standard (24GB) |
| **Complex Reasoning** | Qwen2-VL | 7-72B | Standard to Enterprise |
| **Cost-Sensitive** | Phi-3 Vision | 4.2B | Entry (16GB) |
| **High Accuracy** | LLaVA 1.6 | 7-34B | Standard to Performance |

### Model Size → GPU Requirements

When selecting a model, the parameter count determines the minimum GPU VRAM needed:

| Model Size | VRAM (FP16) | VRAM (INT8) | Recommended GPU Tier | Example Models |
|------------|-------------|-------------|---------------------|----------------|
| **1-3B** | 6-8 GB | 3-4 GB | Entry (16GB) | Small VLMs, embedders |
| **7-8B** | 14-16 GB | 7-8 GB | Entry (16GB) or Standard (24GB) | Llama 3.1 8B, Mistral 7B |
| **13-14B** | 26-28 GB | 13-14 GB | Standard (24GB) or Performance (48GB) | Llama 2 13B |
| **30-34B** | 60-70 GB | 30-35 GB | Performance (48GB) or Enterprise (80GB) | CodeLlama 34B |
| **70B** | 140 GB | 70 GB | Enterprise (80GB) or Multi-GPU | Llama 3.1 70B |
| **100B+** | 200+ GB | 100+ GB | Multi-GPU (2-8x H100) | Large proprietary models |

**Notes:**
- FP16 (half precision) is standard for quality inference
- INT8 quantization reduces VRAM ~50% with minimal quality loss
- Add 20-30% overhead for KV cache during inference
- Clarifai's GPU fractioning allows multiple smaller models on one GPU

### Model Evaluation Phase (Recommended)

Before finalizing the architecture, we recommend a **2-week Model Evaluation Phase**:

1. **Curate Evaluation Dataset** (100-500 representative samples)
2. **Benchmark VLMs**:
   - Start with Clarifai's mm-poly-8b for vision tasks
   - Test 2-3 alternative VLMs for comparison
   - Evaluate accuracy, latency, and cost tradeoffs
3. **Measure Key Metrics**:
   - Accuracy on your specific use cases
   - Latency per request
   - Cost per 1,000 predictions
4. **Document Results** and select best model per use case
5. **Only consider custom training** if VLMs cannot meet requirements

### When Custom Training May Be Needed

Custom model training should only be considered when:
- VLMs cannot achieve required accuracy after prompt optimization
- Latency requirements are <20ms per prediction
- Throughput requirements exceed 500 images/second
- Offline/edge deployment is mandatory
- Cost optimization at massive scale (after validation)
