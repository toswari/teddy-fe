# Agents Overview

This document describes how agents are structured and used in the RapidPrototyping application.

The agents live under `src/agents` and are responsible for specific tasks in the proposal generation
pipeline (discovery, proposal, solution design, etc.). All specialized agents inherit from `BaseAgent`.

---

## Core Classes

### AgentContext

`AgentContext` is a dataclass that carries contextual information into an agent call.

Fields:
- `customer_name: Optional[str]`
- `project_name: Optional[str]`
- `industry: Optional[str]`
- `documents: List[str]` – typically file paths or short descriptors of uploaded docs
- `goals: List[str]` – business or technical goals for the engagement
- `constraints: List[str]` – any limitations (budget, timing, stack, etc.)
- `metadata: Dict[str, Any]` – free-form extra info (e.g., region, priority)
- `conversation_history: List[Message]` – prior chat messages with the LLM

Notes:
- `conversation_history` is appended to on each call to `BaseAgent.generate` when a context is provided.
- Keep `metadata` small and serializable; it may be logged or written out.

### AgentOutput

`AgentOutput` is the structured response from an agent.

Fields:
- `content: str` – the text generated by the agent
- `agent_type: str` – usually the class name of the agent
- `timestamp: datetime` – when the output was produced
- `context: AgentContext` – the context used for the generation
- `metadata: Dict[str, Any]` – extra info such as model/usage/finish_reason

`AgentOutput.save(output_dir: str) -> Path`:
- Creates `output_dir` if it does not exist.
- Writes a Markdown file named `{agent_type}_{YYYYMMDD_HHMMSS}.md`.
- Prepends a YAML-like front matter block with basic metadata (agent, timestamp,
  customer, project, industry) followed by the generated content.

Use this when you want to persist results for later review, sharing, or as
inputs into subsequent stages.

### BaseAgent

`BaseAgent` (in `src/agents/base_agent.py`) is the abstract base class for all
specialized agents.

Constructor signature (simplified):

- `config: Optional[ConfigManager] = None`
- `client: Optional[ClarifaiClient] = None`
- `multimodal_client: Optional[MultimodalClient] = None`

Behavior:
- If `config` is not provided, it calls `get_config()` from `src.config`.
- If `client` is not provided, it constructs a default `ClarifaiClient` with the config.
- Sets `self.agent_name` to the concrete class name.
- Loads a system prompt via `_load_system_prompt()`, which calls
  `config.get_system_prompt(self._get_prompt_file())`.

Required method for subclasses:
- `_get_prompt_file(self) -> str`
  - Must return the relative path (from the config/prompt system) to the system prompt
    used by this agent (e.g., `prompts/discovery.md`).

## Message Building and Generation

### _build_messages

`BaseAgent._build_messages(prompt: str, context: Optional[AgentContext] = None,
include_history: bool = True) -> List[Message]` constructs the message list sent
to the LLM.

Order of messages:
1. **System prompt** (if configured) as `Message(role="system", content=self.system_prompt)`.
2. **Conversation history** from `context.conversation_history` (if `include_history` is True).
3. **Formatted context** (optional):
   - If `context` is provided, `_format_context(context)` is called.
   - When non-empty, a `user` message is added with a preamble
     ("Here is the context for this request:") followed by the formatted context.
   - An `assistant` acknowledgement message is then added:
     "I understand the context. Please provide your request."
4. **Main prompt** as a final `user` message containing the actual request string.

The resulting `messages` list is passed to the `ClarifaiClient.chat` method.

### _format_context

`BaseAgent._format_context(context: AgentContext) -> str` turns the context
into a Markdown-ish string, including:
- Customer, project, industry (when set).
- Bulleted lists of goals and constraints.
- A list of document references.
- Any metadata key/value pairs.

This is how structured context is given to the model in a consistent format.

### generate

`BaseAgent.generate(prompt: str, context: Optional[AgentContext] = None, **kwargs) -> AgentOutput`:

1. Builds messages with `_build_messages`.
2. Calls `self.client.chat(messages, **kwargs)`.
3. If a `context` is provided, appends the user prompt and assistant response
   to `context.conversation_history`.
4. Wraps the response in an `AgentOutput` instance, populating `metadata` with:
   - `usage`
   - `model`
   - `finish_reason`

Typical usage pattern:

```python
from src.agents.discovery_agent import DiscoveryAgent
from src.agents.base_agent import AgentContext

agent = DiscoveryAgent()
context = AgentContext(
    customer_name="Acme Corp",
    project_name="Smart Retail PoC",
    industry="Retail",
    goals=["Improve shelf analytics", "Reduce stock-outs"],
)

output = agent.generate("Generate discovery questions for this engagement.", context=context)
print(output.content)
output_path = output.save("projects/acme-smart-retail/discovery")
print(f"Saved to {output_path}")
```

### stream_generate

`BaseAgent.stream_generate(prompt: str, context: Optional[AgentContext] = None, **kwargs)`
provides a streaming interface:

1. Builds messages with `_build_messages`.
2. Calls `self.client.chat(messages, stream=True, **kwargs)`.
3. Yields each `chunk` from the underlying client.

Example usage:

```python
from src.agents.proposal_agent import ProposalAgent

agent = ProposalAgent()

for chunk in agent.stream_generate("Draft a proposal section for data privacy."):
    print(chunk.content, end="", flush=True)
```

Streaming is useful for UIs (e.g., a web front-end) where partial results
should be shown as soon as they are available.

## Specialized Agents

Concrete agents extend `BaseAgent` and typically only need to:
- Implement `_get_prompt_file` to point at the appropriate system prompt.
- Optionally add helper methods specific to their domain.

Examples (see `src/agents`):
- `DiscoveryAgent` – focuses on generating discovery questions and related assets.
- `ProposalAgent` – generates proposal content based on discovery outputs and context.
- `SolutionAgent` – produces solution architectures, implementation plans, or technical details.

Each of these agents will use its own system prompt (stored under `config/prompts/`)
to specialize behavior while sharing the same core message-building and context-handling logic.

## Programming Notes & Best Practices

- **Context size:**
  - Be mindful of how large `documents`, `goals`, and `constraints` become; very large
    lists may increase token usage. Summarize long documents before adding references.

- **Conversation history:**
  - `context.conversation_history` grows as you call `generate`. For long sessions,
    consider trimming or summarizing older turns.

- **Saving outputs:**
  - Use structured directories under `projects/` (e.g., `projects/<customer>/<project>/...`)
    and pass those paths into `AgentOutput.save`.

- **System prompts:**
  - System prompts live under `config/prompts/`. When adding a new agent, create a
    dedicated prompt file and point `_get_prompt_file` at it.

- **Error handling:**
  - `BaseAgent` assumes `ClarifaiClient.chat` raises on network or API failures.
    Wrap calls to `generate` / `stream_generate` in higher-level error handling
    (e.g., in the web app) as needed.

- **Adding a new agent:**
  1. Create a new file in `src/agents`, e.g., `my_custom_agent.py`.
  2. Subclass `BaseAgent`.
  3. Implement `_get_prompt_file` to point to your system prompt.
  4. (Optional) Add convenience methods that build specific prompts for common tasks.
  5. Wire the agent into the CLI, web app, or orchestration layer as appropriate.

This document should serve as a starting point for understanding and extending
the agent layer in this application.
